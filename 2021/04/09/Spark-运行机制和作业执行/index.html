运行机制和作业执行

执行过程

spark-submit

可指定指定的配置，一般根据作业的性质配置适当的 Executor 的内存和 cpu cores，之后由调度系统(airflow,azkaban等) 进行周期性的调度执行。

命令参数：

  • –master：Standalone 模式时指定集群地址，yarn 模式时指定为 yarn，local 指定为 local, local[N], local[*]
    • –-deploy-mode：运行模式，client 代表 yarn-client，cluster 代表 yarn-cluster
  • –-class： 指定运行的入口类

资源分配参数：

  • --num-executors 2: 指定多少个 executor 进程进行执行
  • --total-executor-cores: 总的核心数量
  • --executor-memory=1G: 每个 executor 使用的内存, PROD 中可以设置为2~4G即可
  • --executor-cores 2： 每个 executor 分配的 CPU
  • --driver-memory 1G: 设置 Driver 进程的内存，使用类似于 collect 之类的 action 算子向 Driver 端拉取数据时，需要调大
  • --driver-cores 1: driver 进程分配多少 core，默认为 1
  • --driver-java-options：Driver 的运行 JVM 参数

覆盖运行的配置参数：

  • –conf：指定运行时候的配置
    • spark.default.parallelism=10: 设置每个 stage 的默认 task 数量，也可以认为是分区数，官网建议设置该参数为 num-executors * executor-cores 的 2~3倍 较为合适
    • spark.storage.memoryFraction: 设置 RDD 持久化数据在 Executor 内存中能占的比例，默认是 0.6
    • spark.shuffle.memoryFraction: 设置 shuffle 过程中一个 task 拉取到上个 stage 的 task 的输出后，进行聚合操作时能够使用的 Executor 内存的比例，默认是0.2。
    • spark.executor.extraJavaOptions: 执行 Executor 的额外 JVM 参数
    • spark.driver.extraJavaOptions=： 指定 Driver 的额外 JVM 参数

其他参数：

  • --jars fastjson.jar,abc.jar： 设置 job 依赖的第三方 jar 包
  • --files: 指定文件位置

通用执行过程

通用的执行过程：

① 使用 spark-submit，用户提交应用 ② 调用用户指定类的 main() 方法，启动 Driver 程序 ③ Driver 程序向 Cluster Manager 要执行 Executor 的资源 ④ Cluster Manager 为 Driver 启动 Executor ⑤ 基于在 RDD 中的 Action 和 Transformation 算子，Driver 发送任务到 Executor ⑥ Executor 执行任务，通过 Cluster Manager 将结果发回到 Driver

Yarn-Client 任务提交

① Driver 端在任务提交的本地机上运行

② Driver 启动之后就会和 ResourceManager 通讯，申请启动一个 ApplicationMaster

③ ResourceManager 会分配 container 容器，在合适的 NodeManager 上启动 ApplicationMaster，负责向 ResourceManager 申请 Executor 内存

④ ResourceManager 接到 ApplicationMaster 的资源申请后会分配 container，然后 ApplicationMaster 在资源分配指定的 NodeManager 上启动 Executor 进程

⑤ Executor 进程启动后会向 Driver 反向注册，Executor 全部注册完成后 Driver 开始执行 main 函数

⑥ 执行到 Action 算子时，触发一个 Job，并根据宽依赖开始划分 stage，每个 stage 生成对应的 TaskSet，之后 Driver 将 task 分发到各个 Executor 上执行

⑦ Executor 执行任务，将执行的结果/状态返回给 Driver

⑧ 应用程序执行完毕，通知 ResourceManager 注销应用，释放资源

Yarn-cluster 提交步骤

Yarn-cluster 运行过程

① Client 先向 ResourceManager 提交请求，并上传 jar 到 HDFS 上

② ResourceManager 在集群中选择一个 NodeManager，在其上启动 AppMaster，在 AppMaster 中实例化 SparkContext(Driver)

③ AppMaster 向 ResourceManager 注册应用程序，注册的目的是申请资源。ResourceManager 监控 App 的运行状态直到结束

④ AppMaster 申请到资源后，与 NodeManager 通信，在 Container 中启动 Executor 进程

⑤ Executor 向 Driver 反向注册，申请任务

⑥ Driver 对应用进行解析，最后将 Task 发送到 Executor 上

⑦ Executor 中执行 Task，并将执行结果或状态汇报给 Driver

⑧ 应用执行完毕，AppMaster 通知 ResourceManager 注销应用，回收资源

WordCount 执行过程

① Master 与 Worker 启动

Master 和 Worker 都为 RPCEntpoint，有对应的生命周期，主要为 onStart, 对于 Master 的 onStart 主要为恢复操作，对于 Worker 的 Start， 向 Master 发送心跳，进行注册

② 应用程序启动(SparkContext 的初始化)

启动三大组件

  • DAGScheduler
  • TaskScheduler
  • SchedulerBackend

向 Master 注册应用程序

命令 Worker 启动 Executor

Executor 向 Driver(应用程序) 注册(反向)，Driver 获取到资源

③ 遇上 Action，触发 Job

④ DAGScheduler.runJob： 根据 RDD 的依赖，从后向前扫描，遇到宽依赖进行 Stage 的划分

⑤ TaskScheduler 获得 Stage(Taskset)，将 Task 放到资源池中

⑥ SchedulerBackend 将 Task 发送到 Executor

⑦ task 在 Executor 中以 pipeline 方式执行。

⑧ 执行 Shuffle Writer，根据不同条件选择 Shuffle 的方式

  • ByPassMergeSortShuffleWriter： …
  • UnSafedShuffleWriter： …
  • SortShuffleWriter： …

⑨ 开始下一个Stage *

  • Shuffle Reader
  • 中间的转换以 pipeline 的方式执行
  • Shuffle Writer

⑩ Executor 执行完毕向 Driver 汇报结果或状态

作业执行

Job 触发

  • Job: 遇到一个 Action 方法则触发一个 Job
  • Stage: 分为 ShuffleMapStage 和 ResultStage
    • ShuffleMapStage，是其他 Stage 的输入，内部的转换操作（map、filter 等）会组成 pipeline，连在一起计算，会产生 map 输出文件
    • ResultStage，一个 job 中只有一个 ResultStage
  • Task: 为 Stage 的子集，以并行度(分区数)来衡量，分区数是多少，则有多少个 task

Stage 划分

由 DAGScheduler 完成

  • DAGScheduler 根据 RDD 的血缘关系构成的 DAG 进行切分，将一个 Job 划分为 若干 Stages，具体划分策略是：从最后一个 RDD 开始，通过回溯依赖判断父依赖 是否是宽依赖（以 Shuffle 为界），划分 Stage；窄依赖的 RDD 之间被划分到同一个 Stage 中，可以进行 pipeline 式的计算
  • 在向前搜索的过程中使用 DFS
  • 最后一个 Stage 称为 ResultStage，其他的都是 ShuffleMapStage
  • 一个 Stage 是否被提交，需要判断它的父 Stage 是否执行。只有父 Stage 执行完毕才能提交当前 Stage，如果一个 Stage 没有父 Stage，那么从该 Stage 开始提交

Task 调度

由 TaskScheduler 来完成，属于底层的调度。DAGScheduler 将 Stage 打包到 TaskSet 交给 TaskScheduler，TaskScheduler 会将 TaskSet 封装为 TaskSetManager 加入到调度队列中。

SchedulerBackend 负责跟外界打交道，接收 Executor 的注册，维护 Executor 的状态。

TaskScheduler 在 SchedulerBackend “问”它的时候，会从调度队列中按照指定的调度策略选择 TaskSetManager 去调度运行

调度策略

由 TaskManager 进行控制，将 DAGScheduler 的 Task 转变为 TaskSetManager 进入到任务队列中，按照一定的策略从中取出来交给选择的 SchedulerBackend 发送到对应的 Executor 执行。

TaskScheduler 以树的方式来管理任务队列，树中的节点类型为 Schedulable，叶子节点为 TaskSetManager，非叶子节点为 Pool。

FIFO 调度策略

FIFO 调度策略是默认模式，在此模式下，只有一个 TaskSetManager 池

Fair 调度策略

该模式下有一个 rootPool 和多个子 Pool，各个子 Pool 中存储着所有待分配的 TaskSetManager；

各个子 Pool，以及子 Pool 中的 TaskSetManager 需要进行排序，Pool 与 TaskSetManager 可使用相同的排序算法。

排序的对象包含三个属性(fairscheduler.xml 配置文件中指定):

  • runningTasks 值（正在运行的 Task 数）
  • minShare 值（时间）
  • weight 值（权重）

本地化调度

本地化调度：

  • PROCESS_LOCAL： 数据和 JVM 代码运行在一起
  • NODE_LOCAL： 数据在相同的节点，如 HDFS 文件在相同的节点上
  • NO_PREF： 可以从任何地方快速访问数据，并且不受位置限制
  • RACK_LOCAL： 数据在同一服务器机架上，数据位于同一机架上的其他服务器上，因此需要通过网络发送
  • ANY： 数据在网络上的其他位置，而不是在同一机架中

Executor 结果返回

对于从 Executor 中获取数据到 Driver 端的结果返回处理：

  • 计算结果在（0，200KB-128MB）区间内：通过 Netty 直接发送给 Driver 终端 Netty 的预留空间 reservedSizeBytes，200KB spark.rpc.message.maxSize，默认值是 128MB
  • 计算结果在 [128MB， 1GB] 区间内：将结果以 taskId 为编号存入到 BlockManager 中，然后通过 Netty 把编号发送给 Driver；阈值可通过 Netty 框架传输参数设置 spark.driver.maxResultSize，默认值 1G
  • 计算结果在（1GB，∞）区间内：直接丢弃，可通过 spark.driver.maxResultSize 配置

失败重试与黑名单机制

提交上报的过程：

对于失败的 Task，会记录它失败的次数，如果失败次数还没有超过最大重试次数， 那么就把它放回待调度的 Task 池子中，否则整个 Application 失败；

在记录 Task 失败次数过程中，会记录它上一次失败所在的 Executor Id 和 Host。下次再调度这个 Task 时，会使用黑名单机制，避免它被调度到上一次失败的节点上，起到一定的容错作用；

配置参数

  • spark.logLineage: 调用 Action 时打印 rdd 的 lineage 信息
  • spark.rpc.message.maxSize： Netty 直接发送给 Driver 终端的阈值，默认 128M
  • spark.driver.maxResultSize： Netty 框架传输参数，默认 1G,大于该值，Executor 向 Driver 发送的数据丢弃
