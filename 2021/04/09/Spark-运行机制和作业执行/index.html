Spark-运行机制和作业执行

运行机制和作业执行

执行过程

spark-submit

可指定 Spark 运行时资源配置，一般根据作业的性质配置适当的 Executor 的内存和 cpu cores，之后由调度系统(airflow, azkaban 等) 进行周期性的调度执行。

命令参数：

  • --master MASTER_URL：

  • Standalone 模式时指定集群地址(spark://host:port)

    • yarn 模式时指定为 yarn
    • local 指定为 local, local[N], local[*]
    • mesos 模式指定地址 mesos://host:port

  • –-class： 指定运行的入口类(Driver)

  • –-deploy-mode：运行模式，决定了 driver 的运行方式，默认为 client

  • --name: 应用名称

  • --num-executors 2: 指定多少个 executor 进程进行执行

  • --total-executor-cores: 总的 executor 核心数量

  • --executor-memory=3G: 每个 executor 使用的内存, PROD 中可以设置为 2~4G

  • --executor-cores 2： 每个 executor 分配的 CPU

  • --driver-memory 1G: 设置 Driver 的内存，使用类似于 collect 之类的 action 算子向 Driver 端拉取数据时，需要调大

  • --driver-cores 1: driver 进程分配多少 core，默认为 1

  • --driver-java-options：Driver 的运行 JVM 参数

覆盖运行的配置参数：

  • --conf ：指定运行时候的配置
  • spark.default.parallelism=10: 设置每个 stage 的默认 task 数量，也可以认为是分区数，官网建议设置该参数为 num-executors * executor-cores 的 2~3倍 较为合适
    • spark.storage.memoryFraction: 设置 RDD 持久化数据在 Executor 内存中能占的比例，默认是 0.6
    • spark.shuffle.memoryFraction: 设置 shuffle 过程中一个 task 拉取到上个 stage 的 task 的输出后，进行聚合操作时能够使用的 Executor 内存的比例，默认是 0.2。
    • spark.executor.extraJavaOptions: 执行 Executor 的额外 JVM 参数
    • spark.driver.extraJavaOptions=： 指定 Driver 的额外 JVM 参数

其他参数：

  • --jars fastjson.jar,abc.jar： 设置 job 依赖的第三方 jar 包
  • --files: 指定文件位置
  • --properties-file FILE： 指定额外的配置文件，默认为 conf/spark-defaults.conf

通用执行过程

通用的执行过程：

1、使用 spark-submit，用户提交应用

2、调用用户指定类的 main() 方法，启动 Driver 程序

3、Driver 程序向 Cluster Manager 要执行 Executor 的资源

4、Cluster Manager 为 Driver 启动 Executor

5、基于在 RDD 中的 Action 和 Transformation 算子，Driver 发送任务到 Executor

6、Executor 执行任务，通过 Cluster Manager 将结果发回到 Driver

spark on yarn

Spark On Yarn 的优势

  • Spark 支持资源动态共享，运行于 Yarn 的框架共享一个集中配置好的资源池
  • 可以很方便的利用 Yarn 的资源调度特性来做分类，隔离以及优先级控制负载，拥有更灵活的调度策略
  • Yarn 可以自由地选择 executor 数量
  • Yarn 是唯一支持 Spark 安全的集群管理器，使用 Yarn，Spark 可以运行于 Kerberized Hadoop 之上，在它们进程之间进行安全认证

Yarn-Client 任务提交

1、Driver 端在任务提交的本地机上运行

2、Driver 启动之后就会和 ResourceManager 通讯，申请启动一个 ApplicationMaster

3、ResourceManager 会分配 container 容器，在合适的 NodeManager 上启动 ApplicationMaster，负责向 ResourceManager 申请 Executor 内存

4、ResourceManager 接到 ApplicationMaster 的资源申请后会分配 container，然后 ApplicationMaster 在资源分配指定的 NodeManager 上启动 Executor 进程

5、Executor 进程启动后向 Driver 反向注册，Executor 全部注册完成后 Driver 开始执行 main 函数

6、执行到 Action 算子时，触发一个 Job，并根据宽依赖开始划分 stage，每个 stage 生成对应的 TaskSet，之后 Driver 将 task 分发到各个 Executor 上执行

7、Executor 执行任务，将执行的结果/状态返回给 Driver

8、应用程序执行完毕，通知 ResourceManager 注销应用，释放资源

Yarn-cluster 提交步骤

1、Client 先向 ResourceManager 提交请求，并上传 jar 到 HDFS 上

2、ResourceManager 在集群中选择一个 NodeManager，在其上启动 AppMaster，在 AppMaster 中实例化 SparkContext(Driver)

3、AppMaster 向 ResourceManager 注册应用程序，注册的目的是申请资源。ResourceManager 监控 App 的运行状态直到结束

4、AppMaster 申请到资源后，与 NodeManager 通信，在 Container 中启动 Executor 进程

5、Executor 向 Driver 反向注册，申请任务

6、Driver 对应用进行解析，最后将 Task 发送到 Executor 上

7、Executor 中执行 Task，并将执行结果或状态汇报给 Driver

8、应用执行完毕，AppMaster 通知 ResourceManager 注销应用，回收资源

WordCount 执行过程

1、Master 与 Worker 启动

Master, Worker 都为 RPCEntpoint，有对应的生命周期(construct → onStart → receive* → onStop),对于 Master 的 onStart 主要为恢复操作，对于 Worker 的 onStart，向 Master 发送心跳，进行注册

2、 应用程序启动(SparkContext 的初始化)

启动任务调度组件

  • DAGScheduler
  • TaskScheduler
  • SchedulerBackend

向 Master 注册应用程序，命令 Worker 启动 Executor，Executor 向 Driver 注册(反向)，Driver 获取到资源

3、 执行 driver 的 main 方法，遇上 Action，触发 Job

4、 DAGScheduler.runJob： 根据 RDD 的依赖，从后向前扫描，遇到宽依赖进行 Stage 的划分

5、 TaskScheduler 获得 Stage(Taskset)，将 Task 放到资源池中

6、 SchedulerBackend 将 Task 发送到 Executor

7、 task 在 Executor 中以 pipeline 方式执行。

8、 执行 Shuffle Writer，根据不同条件选择 Shuffle 的方式

  • ByPassMergeSortShuffleWriter： 无 mapSideCombine, 分区数少于200(..)…
  • UnSafedShuffleWriter：序列化方式支持 relocation(kyro..)，分区数小于 2^24,..
  • SortShuffleWriter： 其他的

9、 开始下一个Stage *

  • Shuffle Reader
  • 中间的转换以 pipeline 的方式执行
  • Shuffle Writer

10、 Executor 执行完毕向 Driver 汇报结果或状态

作业执行

整体的执行过程：

dagScheduler.submitJob 发生阻塞，直到作业完成，Job 为串行提交的

dag.submitMisssingTasks: 没有父依赖时的调用

Job 触发

  • Job: 遇到一个 Action 方法则触发一个 Job，通过 dagScheduler.submitJob 为串行提交，阻塞直到作业完成/失败。
  • Stage: ShuffleMapStage 和 ResultStage
    • ShuffleMapStage： 其他 Stage 的输入，内部的转换操作会组成 pipeline，连在一起计算，会产生 map 输出文件
    • ResultStage： 一个 job 中只有一个 ResultStage
  • Task: 为 Stage 的子集，以并行度来衡量，分区数是多少，则有多少个 task

Stage 划分

由 DAGScheduler 完成

  • DAGScheduler 根据 RDD 的血缘关系构成的 DAG 进行切分，将一个 Job 划分为若干 Stages，具体划分策略是：从最后一个 RDD 开始，通过回溯依赖(DFS)判断父依赖是否是宽依赖（以 Shuffle 为界），划分 Stage；窄依赖的 RDD 之间被划分到同一个 Stage 中，可以进行 pipeline 式的计算
  • 最后一个 Stage 称为 ResultStage，其他的都是 ShuffleMapStage
  • 一个 Stage 是否被提交，需要判断它的父 Stage 是否执行。只有父 Stage 执行完毕才能提交当前 Stage，如果一个 Stage 没有父 Stage，那么从该 Stage 开始提交

Task 调度

由 TaskScheduler 来完成，属于底层的调度。DAGScheduler 将 Stage 打包到 TaskSet 交给 TaskScheduler，TaskScheduler 会将 TaskSet 封装为 TaskSetManager 加入到调度队列中。SchedulerBackend 负责跟外界打交道，接收 Executor 的注册，维护 Executor 的状态。

TaskScheduler 在 SchedulerBackend “问”它的时候，会从调度队列中按照指定的调度策略选择 TaskSetManager 去调度运行

Task 的分类：

  • ShuffleMapTask: 生成的是 MapStatus, 执行任务，并将任务输出划分到多个桶(基于任务的分区)
  • ResultTask: 生成的 result 是 func 在 partition 上的执行结果, 执行该 Task 并将 Task 输出发送回驱动程序应用程序

调度策略

由 TaskManager 进行控制，将 DAGScheduler 的 Task 转变为 TaskSetManager 进入到任务队列中，按照一定的策略从中取出来交给选择的 SchedulerBackend 发送到对应的 Executor 执行。TaskScheduler 以树的方式来管理任务队列，树中的节点类型为 Schedulable，叶子节点为 TaskSetManager，非叶子节点为 Pool。

相关的调度策略：

  • FIFO 调度策略： FIFO 调度策略是默认模式，在此模式下，只有一个 TaskSetManager 池

  • Fair 调度策略：

    该模式下有一个 rootPool 和多个子 Pool，各个子 Pool 中存储着所有待分配的 TaskSetManager。先对子 Pool 排序，再对子 Pool 里面的 TaskSetManager 进行排序，Pool 和 TaskSetManager 可使用相同的排序算法。

    • 排序的对象包含三个属性(fairscheduler.xml配置文件中指定):

      • runningTasks 值： 正在运行的 Task 数。为动态的值，程序中从 taskSet 中获取
      • minShare 值： 时间
      • weight 值： 权重

    • 让 minShare 使用率和权重使用率少(运行 task 较少)的先运行，排序完成后放入 ArrayBuffer。

    • 取出 TaskSetManager 后，按照一定规则取出 Task 给 TaskScheduler，TaskScheduler 交给 SchedulerBackend，最终分发给 Executor 上执行。

本地化调度

DAGScheduler 中的 submitMissingTasks 确定每个需要计算的 Task 的 preferredLocations，getPreferrdeLocations 得到分区的优先位置，一个 partition 对应一个 Task，该 Partition 的优先位置对应 Task 的优先位置。

根据 Task 的优先位置，确定 Task 的 Locality 级别。调度执行时，总是尽量让每个 Task 以最高的本地性级别来启动。

按照某个本地性级别启动，对应的节点没有空闲导致启动失败，在指定时间内再次尝试以对应的本地性级别启动，若超时则降级启动，尝试下一个本地性级别。

相关的类：

  • TaskLocation： Task 的位置，由 DAGScheduler 中的 submitMissingTasks 计算出来，当前有三种实现 HostTaskLocation、HDFSCacheTaskLocation、HostTaskLocation

本地化调度级别枚举项：

  • PROCESS_LOCAL： 数据和 JVM 代码运行在一起
  • NODE_LOCAL： 数据在相同的节点，如 HDFS 文件在相同的节点上
  • NO_PREF： 可以从任何地方快速访问数据，并且不受位置限制
  • RACK_LOCAL： 数据在同一服务器机架上， 需要通过网络发送
  • ANY： 数据在网络上的其他位置，而不是在同一机架中

相关的 Task 本地性等待时间：

  • spark.locality.wait：默认的等待时长，默认为 3s
  • spark.locality.wait.process： PROCESS_LOCAL 的等待时长
  • spark.locality.wait.node： NODE_LOCAL 的等待时长
  • spark.locality.wait.rack： RACK_LOCAL 的等待时长

相关的 TaskFailedReason:

  • FetchFailed: Task 从远端节点获取 shuffle 数据失败，会被标记为成功，不会重新运行，缺少 map 数据需要首先重新生成
  • TaskResultLost: Task 成功完成，但是结果在 executor 的 BlockManager 取回之前就丢失了
  • UnknownReason: 未知的 …
  • ..

Executor 结果返回

从 Executor 中获取数据到 Driver 端根据不同的大小进行处理：

  • result size ∈ (0，200KB-128MB)： 通过 Netty 直接发送给 Driver 终端，Netty 的预留空间 reservedSizeBytes，200KB spark.rpc.message.maxSize，默认值是 128MB
  • result size ∈ [128MB， 1GB]：将结果以 taskId 为编号存入到 BlockManager 中，然后通过 Netty 把编号发送给 Driver；阈值可通过 Netty 框架传输参数设置 spark.driver.maxResultSize，默认值 1G
  • result size ∈ (1GB，∞)：直接丢弃，可通过 spark.driver.maxResultSize 配置

失败重试与黑名单机制

提交上报的过程：

对于失败的 Task，会记录失败的次数，如果失败次数还没有超过最大重试次数， 那么就把它放回待调度的 Task 池子中，否则整个 Application 失败；

在记录 Task 失败次数过程中，会记录它上一次失败所在的 Executor Id 和 Host。下次再调度这个 Task 时，会使用黑名单机制，避免它被调度到上一次失败的节点上，起到一定的容错作用；

任务调度组件

  • DAGScheduler： class，负责高层的调度，将 Job 转换为 DAG graph
  • TaskScheduler： trait, 负责实际每个具体 Task 的物理调度执行，默认只有一种实现 TaskSchedulerImpl
    • 确定公共的逻辑，如决定作业执行的调度顺序，唤醒并启动推测任务
  • SchedulerBackend：trait, 有多种实现，分别对应不同的资源管理器, 完成应用程序向 Master 的注册

DAGScheduler

创建 Job, 将 DAG 中的 RDD 划分到不同的 Stage，提交 Stage 等

成员变量：

  • JobWaiter: 等待 DAGScheduler 的 Job 完成，每个 Task 结束，通过回调函数将结果传递给 resultHandler 处理
  • DAGSchedulerEventProessLoop: 内部的事件循环处理器，处理 DAGSchedulerEvent 类型的事件
    • EventLoop： 消息异步处理策略类
    • DAGSchedulerEvent: 与 DAG 调度相关的事件，包含如下的事件
      • JobSubmitted: 在目标 RDD 上提交结果生成 Job
      • MapStageSubmitted： 提交 Map 阶段
      • BeginEvent: Task 开始调度，由 TaskManager 的 resourceOffer 中调用
      • GettingResultEvent： Task 的结果获取事件，由 TaskManager 的 handleTaskGettingResult 调用
      • CompletionEvent： Task 完成事件
      • TaskSetFailed： 整个 TaskSet 失败，由 TaskSetManager 的 abort 调用
      • ResubmitFailedStages: 重新提交失败的 Stage
      • SpeculativeTaskSubmitted：推测 Task 提交事件，由 TaskSetManager 的 checkSpeculatableTasks 方法调用
      • …
  • LiveListenerBus： 异步传递 SparkListenerEvents 到注册的 sparkListener
    • SparkListenerEvent： 相比 DAGSchedulerEvent 所拥有的事件种类更多
      • SparkListenerStageSubmitted： Stage 提交事件
      • SparkListenerStageCompleted： Stage 完成事件
      • SparkListenerTaskStart： Task 启动事件
      • SparkListenerTaskGettingResult: Task 获取结果事件
      • SparkListenerSpeculativeTaskSubmitted: 推测任务提交事件
      • SparkListenerTaskEnd: Task 结束事件
      • SparkListenerJobStart: Job 启动事件
      • SparkListenerBlockManagerAdded： BlockManager 添加事件
      • …

主要的方法：

  • runJob： 提交 job，发生阻塞，串行执行，内部调用 submitJob

  • submitJob

TaskSetManager 源码

  • TaskSetManager 跟踪每个任务，在任务失败时重试任务(最多是有限的次数)
  • 调度策略： 将 TaskSetManager 放到任务队列后的处理
    • FIFO: TaskSetManager 根据先后顺序进入队列，出队列时取先进入的，为默认策略，只有一个 TaskSetManager 池。
    • Fair： 先对子 Pool 进行排序，再对子 Pool 中的 TaskSetManager 进行排序
  • 黑名单追踪
  • 推测执行(speculation)： 使用 MedianHeap 获取成功 Task 的执行持续时间中位数

相关方法：

  • resourceOffer: 询问 TaskSet 是否想在一个 node 上运行任务
  • statusUpdate: 告诉 TaskSet 它的一个 Task 改变了状态(例如完成)

SchedulerBackend

一个 trait, 调度系统的后端接口，允许在 TaskSchedulerImpl 下插入不同的系统。有多种实现，如 StandaloneSchedulerBackend、LocalSchedulerBackend、CoarseGrainedSchedulerBackend。

  • reviveOffers： 在 TaskSchedulerImpl 的 submitTasks(taskSet) 中调用

CoarseGrainedSchedulerBackend

粗粒度的调度方式

  • start: 对内部类 driverEndpoint 进行注册
  • makeOffers： 将集群的资源以 Offer 的方式发给上层的 TaskScheduler 处理。TaskSchedulerImpl 调用 scheduler.resourceOffers 获得要被执行的 Seq[TaskDescription]，然后将得到的 Seq[TaskDescription] 交给 CoarseGrainedSchedulerBackend 分发到各个 executor 上执行
  • DriverEndpoint：
    • onStart: 创建定时调度任务，定时发送 ReceiveOffers 消息，最终调用 makeOffers 方法处理

配置参数

  • spark.executor.extraJavaOptions executor 额外的 jvm 参数
  • spark.executor.memory： executor 内存大小，默认 1g
  • spark.task.maxFailures： task 最大尝试失败的次数
  • spark.blacklist.enabled： 是否开启黑名单
  • spark.rpc.message.maxSize： Netty 直接发送给 Driver 终端的阈值，默认 128M
  • spark.driver.maxResultSize： Netty 框架传输参数，默认 1G,大于该值，Executor 向 Driver 发送的数据丢弃
  • spark.logLineage: 调用 Action 时打印 rdd 的 lineage 信息, 调试时使用

Task 本地性的超时降级配置：

  • spark.locality.wait：默认的等待时长，默认为 3s
  • spark.locality.wait.process： PROCESS_LOCAL 的等待时长
  • spark.locality.wait.node： NODE_LOCAL 的等待时长
  • spark.locality.wait.rack： RACK_LOCAL 的等待时长

fairscheduler.xml： TaskScheduler 选择 FAIR 调度策略时候的参数

  • minShare：
  • weight： 权重
