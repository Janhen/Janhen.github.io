Kafka-基础

Created2021-04-09|Updated2021-04-15|大数据
|Post View:

概述

一个分布式、分区的、多副本的、多生产者、多订阅者，基于 zookeeper 协调的分布式日志系统, 常见可以用于 web/nginx 日志、访问日志，消息服务等。

消息中间件的模式，只有消息的拉取，没有推送，可通过轮询实现消息的推送。

消息被分批写入 Kafka。 批次为一组消息，消息属于同一个主题和分区。可对批次进行压缩。吞吐量大。

优点

持久化存储： 零拷贝、顺序读与顺序写、Linux 的页缓存。

流平台关键功能：

  • 消息队列：发布和订阅消息流，这个功能类似于消息队列，这也是 Kafka 也被归类为消息队列 的原因。
  • 容错的持久⽅式存储记录消息流： Kafka 会把消息持久化到磁盘，有效避免了消息丢失的⻛险。
  • 流式处理平台： 在消息发布的时候进⾏处理，Kafka 提供了⼀个完整的流式处理类库。

主要应⽤场景：

  • 消息队列：建⽴实时流数据管道，以可靠地在系统或应⽤程序之间获取数据。
  • 数据处理：构建实时的流数据处理程序来转换或处理数据流。
    • 用户活动跟踪
    • 运营指标
    • 流式处理
    • 日志、消息

Kafa 设计中的两个反常识：

  • 消息使用磁盘存储，非内存
  • 分区副本保持数据一致性(高可用保证)，非过半机制，随机选择一个 ISR，依据配置可选一个 USR。

Kafka 的扩展性： 可对生产者、消费者、 Brokder、主题的分区进行横向扩展。

consumer 采用 pull 模式从 brokder 中读取数据，可自主控制消费消息的速率, 可控制消费方式..

推送, 保证及时性, RabbitMq 保证 um, 小而美。

架构图

  • Producer: 生产者
    • 给定 key, 无 key, 自定义分区器
    • 生产端通过轮询、哈希(基于 Key)、指定放到特定分区(基于分区号)
  • Consumer: 消费者
    • 再平衡, 如某个 Consumer 挂掉，对多个分区进行重新分配。
    • 可横向扩展，需要保证不能重复消费。一个分区只能被一个消费组中的一个消费者消费。
    • Consumer Group: 同一个消费组中的消费者不会对 Topic 的消息进行重复消费。
  • Broker: 可看作为一个 Kafka 实例，多个 Broker 组成 Kafka 分布式集群
    • 对于 topic 的 N partition，分区数不能大于 broker 的数量
    • 集群控制器：特殊的 Brokder, 用于分区的分配、监控 broker
    • Broker 与集群： 单个 Broker 可处理多个分区和每秒百万级的消息量。
  • Topic: 类似 DB 的表，分库分表的逻辑表。
  • Partiton: 主题实际存储的分片, 无法保证主题范围内消息的顺序，通过分区实现数据冗余与伸缩。
  • Replicas：首领副本、追随者副本。主题分区的，不让相同分区的副本在同一台机器上。
    • AR(Assigned Replicas): 分区的所有副本为
    • ISR(In-Sync Replicas): 同步副本
    • OSR(Out-Sync Replicas): 不同步副本
  • Offset：递增序列，可指定从 0 开始消费，或者指定的偏移量进行消费。
    • 偏移量 HW: 决定对外提供消息的最大偏移量，再 HW 前可见、可消费，之后不允许消费
    • 下一条待写入消息的 offset(LEO):
  • Event(Message): 生产者发送的消息，需要指定 topic、key、分区、消息内容

对比其他 MQ 的比较

Kafka 相⽐其他消息队列主要的优势如下：

  • 极致的性能：基于 Scala 和 Java 语⾔开发，设计中⼤量使⽤了批量处理和异步的思想，最⾼可以每秒处理千万级别的消息。
  • ⽣态系统兼容性高：Kafka 与周边⽣态系统的兼容性好，尤其在⼤数据和流计算领域。

与 RabbitMQ, Rocket MQ 相比，吞吐量大，延迟高，提供的功能少，如相对于 RabbitMQ 没有提供默认的 RetryQueue。

执行过程

生产者执行原理

两个线程：

① 主线程：负责消息创建，拦截器，序列化器，分区器等操作，并将消息追加到消息收集器 RecoderAccumulator 中

② Sender 线程：

生产者发送过程

① Producer 创建时，会创建一个 Sender 线程并设置为守护线程 ② Producer 生产消息时，内部是异步流程；生产的消息先经过拦截器->序列化器->分区器，然后将消息缓存在缓冲区（该缓冲区也是在 Producer 创建时创建）。 ③ 发送批次消息，批次发送的条件为：缓冲区数据大小达到 batch.size 或者 linger.ms 达到上限。 ④ 批次发送后，发往指定分区，然后落盘到 broker；如果生产者配置了 retrires 参数大于 0 并且失败原因允许重试，那么 Client 内部会对该消息进行重试。 ⑤ 落盘到 broker 成功，返回生产元数据给生产者。 ⑥ Producer 根据元数据进行处理，元数据返回有两种方式：一种是通过阻塞直接返回，另一种是通过回调返回。

序列化器

  • Kyro 序列化： Spark 中常用的。
  • 自定义序列化器需要实现 org.apache.kafka.common.serialization.Serializer<T> 接口，并实现其 中的 serialize 方法。

Offset 提交

kafka 会定期把 group 消费情况保存起来，做成一个 offset map。

有默认的主题 __consumer_offsets 控制偏移量，位移提交有同步和异步提交。

(1) 自动提交

可能导致重复消费

自动提交不会出现消息丢失，但可能会重复消费。

(2) 异步提交

异步提交不会进行自动重复，推荐的方式。

大部分进行异步，每隔一段时间/数据量进行一次同步提交。

offset 特殊场景

消费者第一次消费数据，或者消费者查找之前保留的 offset 没查到时候的处理

auto.offset.reset：

  • earliest：从最早的数据开始消费
  • latest(default): 从最新的数据开始消费
  • none：根据指定的 group.id 未找到之前消费的 offset 信息，抛异常

消费组管理

消费组的状态

  • Dead：组内已经没有任何成员的最终状态
  • Empty：组内无成员
  • PreparingRebalance：组准备开启新的rebalance，等待成员加入
  • AwaitingSync：正在等待 leader consumer 将分配方案传给各个成员
  • Stable：再均衡完成

分区

为了做到均匀分布，通常 partition 的数量通常是 Broker Server 数量的整数倍。

副本机制

  • 分区中会有多个副本，其中一个副本为 Leader 其他的副本为 Follower。发送消息的时候会被发送到 Leader，之后 Follower 副本会从 Leader 副本中拉取消息进行同步。
  • Producer 和 Consumer 只和 Leader 副本进行交互，其他 Follower 副本为 Leader 的拷贝，用于保证消息存储的安全性，只有 Leader 副本发生故障时使用，从 Follower 中徐安全一个 Leader。
  • Follower 副本中的数据可能和 Leader 副本不同步，在 Leader 不可用时先跳过不同步的 Follower 副本，即 USR。
  • Follower 分区和普通 Kafka 消费者一样，消费 Leader 分区消息，允许 Follower 进行批处理，并将其持久化到自己的日志。

Kafka 分区和多副本机制的好处

  • 给特定的 Topic 指定多个 Partition，各个 Partition 分布在不同的 Broker 上，提供很好的并发能力
  • Partition 可指定对应的副本数量，提高了消息存储的安全性，提高了容灾能力

分区副本在 Broker 上的存放策略

(1) 不考虑机架的分配方案：

第一轮顺序放

第二轮，Partiton1 –> Brokder1，Partition2 –> Brokder2，Partiton3 –> Broder3

第三轮, Partiton1 –> Brokder, Partition2 –> Brokder3, Partiton3 –> Brokder4

(2) 考虑机架的分配情况

按照基本规则分配，达到特定情况区分于基本规则配置，防止某几个 Broker 总是出现问题。

出现问题时选取 Partition 的副本

此处由集群控制器控制，判断条件如下：

① 有 ISR 存在，随机找一个 ISR 进行恢复

② 所有的 ISR 都宕机，等一个 ISR 恢复

③ 所有的 ISR 都宕机，在配置可选择 OSR 时，选择一个 OSR 作为 Leader

向已经部署好的 Kafka 集群里面添加机器，从已经部署好的 Kafka 节点中复制相应的配置文件，把里面的 broker id 修改成全局唯一的，最后启动这个节点即可将它加入到现有 Kafka 集群中。 ⇒ 新添加的 Kafka 节点并不会自动地分配数据，无法分担集群的负载

分区与消费者的再分配

再平衡的时候无法进行消费，导致消费能力下降。

触发条件：

① 消费者组内成员发生变更： 增减消费者

② 主题的分区数发生变更： 增加主题的分区

③ 订阅的主题发生变化： 消费组通过正则订阅主题, 对应消费组订阅的 Topic 数量变更

消费者进行横向扩展

分区不能被多个消费者消费, Topic 的分区越多，越有利于消费组的扩展。

心跳机制

Kafka 的心跳是 Kafka Consumer 和 Broker 之间的健康检查，只有当 Broker Coordinator 正常时，Consumer 才会发送心跳。

控制 Topic 的分区与消费组中的消费者对应关系。

再平衡的影响：

重平衡过程中，消费者无法从 kafka 消费消息，对 kafka 的 TPS 影响极大，重平衡可能会耗时极多，这段时间 kafka 基本处于不可用状态。

再平衡的过程

在协调器收集到所有成员请求前，它会把已收到请求放入一个叫 purgatory(炼狱)的地方。

消费组的分区分配方案在客户端执行。Kafka 交给 Client 可以有更好的灵活性。

分区分配策略

Kafka 默认采用 RangeAssignor 分配算法

1、RangeAssignor

偏向于字典序小的消费组中的消费者比较 “贪婪”

2、RoundRobinAssignor

轮询的方式进行指定。

3、StickyAssignor

分区的分配尽量的均衡，每一次重分配的结果尽量与上一次分配结果保持一致,和 JDK8 中 HashMap 的 rehash 对低位的不变相同思想。

4、自定义分配策略

可控制再平衡的三个参数

  • session.timeout.ms: 控制心跳的超时时间, 6s
  • heartbeat.interval.ms: 心跳的频率, 2s, 一般为上面值的三分之一
  • max.poll.interval.ms: poll 的间隔, 1 分钟

其他

Zookeeper 作用

存储 Kafka 中的元数据，在 Kafka 2.8+ 后移除，不再使用

ZK 为 Kafka 提供的功能:

  • Broker 注册： 有一个记录 Broker 服务器列表的节点，每次 Broker 启动时，都会到 ZooKeeper 上进行注册，写数据到 brokers/ids 下新建节点，将 Broker 中的 IP 地址和端口信息记录进去。
  • Topic 注册：同一个 Topic 被分成多个分区分布到多个 Broker 中，分区信息与 Broker 的对应关系由 ZooKeeper 维护。对应到文件夹 brokers/topics//Partitons/ 新建节点。
  • 负载均衡：Kafka 通过特定 Topic 指定多个 Partition，各个 Partition 分布到不同的 Broker 中，提供较好的并发能力。对同一个 Topic 的多个 Partition，尽量将 Partition 分布到不同的 Broker 上，生产者投递消息到不同的 Broker 的 Partition 中。当 Consumer 消费时，ZooKeeper 根据当前 Partition 数量和 Consumer 数量实现动态负载均衡。
